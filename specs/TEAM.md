# Team Charter — Game Trend AI Team

## 미션

게임 운영 인원(마케팅·사업·기획·CS)이 **이상 징후를 직접 파악하지 않아도**,  
AI가 먼저 감지하고 원인을 분석하여 대응 방안을 밀어주는  
**능동적 AI 업무 비서**를 제작한다.

---

## 팀 구성

| 역할 | 담당 모듈 | 상태 |
|------|-----------|------|
| **팀장** (Claude) | 설계·리뷰·통합 | 상시 |
| **Agent A** | `backend/detector/` — 이상 감지 엔진 | 완료 |
| **Agent B** | `backend/notifier/` — Slack 알림 | 예정 |
| **Agent C** | `backend/analyzer/action_recommender.py` — 대응 제안 | 완료 |
| **Agent D** | `backend/api/alerts.py` + `frontend/` — API·UI | 예정 |

## 운영 원칙

1. **계약 우선**: 모듈 간 인터페이스(입력·출력 스키마)는 구현 전에 문서로 확정한다.  
   구현 중 계약을 바꿔야 하면 팀장 검토 후 문서를 먼저 수정한다.

2. **알림 품질 > 알림 수량**: 오탐(false positive)은 팀 신뢰를 깎는다.  
   임계값을 낮추는 변경은 반드시 근거 데이터와 함께 제안한다.

3. **장애 격리**: 한 에이전트의 실패가 전체 파이프라인을 멈추지 않는다.  
   각 모듈은 예외를 삼키고 로깅한 뒤 다음 게임으로 넘어간다.

4. **DB는 단일 진실 공급원**: 에이전트 간 상태 공유는 DB를 통해서만 한다.  
   인메모리 공유 상태, 전역 변수 금지.

5. **멱등성 보장**: 동일 입력으로 중복 실행해도 DB 상태가 변하지 않아야 한다.  
   (쿨다운 체크, upsert 패턴 활용)

## 에이전트 간 협업 프로토콜

```
크롤링 완료
  └─▶ Agent A: 이상 감지
        └─▶ Alert 생성 (DB flush)
              └─▶ Agent C: 대응 제안 생성 (Claude API)
                    └─▶ Alert.recommendations 저장 (DB commit)
                          └─▶ Agent B: Slack 알림 전송
```

- **A → C**: Alert 객체 + 현재 윈도우 posts를 인수로 직접 전달 (같은 트랜잭션)
- **A+C → B**: DB commit 완료 후 Alert ID를 넘겨 비동기 전송
- **B → D**: DB 폴링 또는 FastAPI 엔드포인트로 읽기 (단방향)

## 품질 기준

| 항목 | 기준 |
|------|------|
| 오탐율 | CRITICAL 알림의 오탐 < 20% (운영 2주 후 리뷰) |
| 알림 지연 | 크롤링 완료 후 Slack 수신까지 < 2분 |
| 대응 제안 품질 | 부서별 액션 아이템이 구체적이고 즉시 실행 가능한 수준 |
| API 응답 | 95th percentile < 500ms |
